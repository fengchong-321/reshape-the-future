```
7. 应用层    → HTTP、FTP、SMTP、DNS

1. 网络接口层（对应 OSI 1-2）
```

### 2.2 TCP 三次握手与四次挥手

```
三次握手（建立连接）：

客户端                    服务端
   |                         |
   |------- SYN ------------>| (客户端请求建立连接)
   |                         |
   |<---- SYN + ACK ---------| (服务端确认并请求)
   |                         |
   |------- ACK ------------>| (客户端确认)
   |                         |
   [连接建立]

为什么是三次？
- 防止已失效的连接请求突然到达服务端
- 确保双方都能确认对方的发送和接收能力

四次挥手（断开连接）：

客户端                    服务端
   |                         |
   |------- FIN ------------>| (客户端请求关闭)
   |                         |
   |<------ ACK -------------| (服务端确认)
   |                         |
   |                         | (服务端可能还有数据发送)
   |                         |
   |<------ FIN -------------| (服务端请求关闭)
   |                         |
   |------- ACK ------------>| (客户端确认)
   |                         |
   [连接关闭]

为什么是四次？
- TCP 是全双工，每个方向的关闭需要单独确认
- 服务端收到 FIN 后可能还有数据要发送

TIME_WAIT 状态：
- 客户端发送最后一个 ACK 后进入
- 持续 2MSL（最大报文生存时间）
- 作用：
  1. 确保最后的 ACK 到达服务端
  2. 让旧连接的报文在网络中消失
```

**面试常见问题：**

```
Q1: 为什么三次握手不是两次？

A:
1. 防止已失效的 SYN 报文突然到达服务端
2. 确保双方都能确认对方的发送和接收能力
3. 两次握手无法防止这种情况：
   客户端发送的 SYN 在网络中滞留
   客户端超时重发并完成连接
   旧 SYN 到达服务端，服务端误认为新连接

Q2: 为什么四次挥手不是三次？

A:
TCP 是全双工通信：
1. 客户端 FIN 表示"我没数据了"，但仍可接收
2. 服务端 ACK 表示"知道了"
3. 服务端可能还有数据要发送
4. 服务端 FIN 表示"我也没数据了"
5. 客户端 ACK 确认

如果服务端没数据要发，可以合并 ACK 和 FIN，变成三次。

Q3: TIME_WAIT 过多怎么办？

A:
原因：大量短连接
影响：占用端口资源（最多 65535 个）
解决：
1. 开启 SO_REUSEADDR
2. 调整内核参数（tcp_tw_reuse）
3. 使用长连接
4. 负载均衡分散连接
```

### 2.3 TCP vs UDP

```
TCP（传输控制协议）：
- 面向连接
- 可靠传输（确认、重传、流量控制、拥塞控制）
- 有序到达
- 全双工
- 面向字节流
- 开销大

UDP（用户数据报协议）：
- 无连接
- 不可靠（不保证到达、不保证顺序）
- 面向报文
- 开销小
- 支持一对多

应用场景：

TCP：
- HTTP/HTTPS
- FTP
- SMTP
- SSH
- 需要可靠传输的场景

UDP：
- DNS
- 视频直播
- 在线游戏
- VoIP
- 需要实时性的场景

TCP 可靠性保证：
1. 序号和确认号
2. 超时重传
3. 滑动窗口（流量控制）
4. 拥塞控制（慢启动、拥塞避免、快重传、快恢复）
```

**面试常见问题：**

```
Q1: 什么时候用 TCP，什么时候用 UDP？

A:
TCP：
- 需要数据完整性（文件传输、邮件）
- 不能容忍丢包
- 实时性要求不高

UDP：
- 需要实时性（视频、游戏）
- 可以容忍少量丢包
- 局域网环境

Q2: TCP 如何保证可靠性？

A:
1. 校验和：检测数据损坏
2. 序号/确认号：保证有序和到达
3. 超时重传：未收到确认则重传
4. 滑动窗口：流量控制
5. 拥塞控制：避免网络拥塞

Q3: 什么是粘包/拆包？

A:
原因：TCP 面向字节流，没有消息边界

粘包：多个小消息合并成一个
拆包：一个大消息分成多个

解决：
1. 固定长度
2. 分隔符
3. 长度字段（最常用）
```

### 2.4 HTTP/HTTPS

```
HTTP 特点：
- 无状态（每次请求独立）
- 基于请求-响应
- 简单快速
- 灵活（可传输任意类型数据）

HTTP 版本：
- HTTP/1.0：每次请求新建连接
- HTTP/1.1：持久连接、管道化、分块传输
- HTTP/2：多路复用、头部压缩、服务器推送
- HTTP/3：基于 QUIC（UDP）

HTTP 状态码：
1xx：信息性
2xx：成功
  - 200 OK
  - 201 Created
  - 204 No Content
3xx：重定向
  - 301 永久重定向
  - 302 临时重定向
  - 304 Not Modified
4xx：客户端错误
  - 400 Bad Request
  - 401 Unauthorized
  - 403 Forbidden
  - 404 Not Found
5xx：服务端错误
  - 500 Internal Server Error
  - 502 Bad Gateway
  - 503 Service Unavailable

HTTP 方法：
- GET：获取资源
- POST：创建资源
- PUT：更新资源（完整）
- PATCH：更新资源（部分）
- DELETE：删除资源
- HEAD：获取头部
- OPTIONS：获取支持的方法

GET vs POST：
1. 参数位置：GET 在 URL，POST 在请求体
2. 长度限制：GET 有限制（URL 长度），POST 无限制
3. 缓存：GET 可缓存，POST 不可
4. 安全性：都不安全（需 HTTPS）
5. 幂等性：GET 幂等，POST 不幂等
```

**HTTPS：**

```
HTTPS = HTTP + TLS/SSL

加密过程：
1. 客户端发起请求
2. 服务端返回证书（含公钥）
3. 客户端验证证书
4. 客户端生成对称密钥，用公钥加密发送
5. 服务端用私钥解密获取对称密钥
6. 双方用对称密钥通信

为什么用对称+非对称？
- 非对称加密慢，用于交换密钥
- 对称加密快，用于数据传输

HTTPS 作用：
1. 加密：防止窃听
2. 认证：验证服务器身份
3. 完整性：防止篡改
```

**面试常见问题：**

```
Q1: HTTP 和 HTTPS 区别？

A:
1. 安全性：HTTPS 加密传输
2. 端口：HTTP 80，HTTPS 443
3. 证书：HTTPS 需要 CA 证书
4. 性能：HTTPS 略慢（握手开销）
5. 成本：HTTPS 需要购买证书

Q2: HTTPS 如何保证安全？

A:
1. 证书验证：确保连接正确的服务器
2. 非对称加密：安全交换对称密钥
3. 对称加密：加密通信内容
4. 消息摘要：保证数据完整性

Q3: HTTP/2 相比 HTTP/1.1 的改进？

A:
1. 多路复用：一个连接并行多个请求
2. 头部压缩：HPACK 压缩算法
3. 二进制分帧：更高效解析
4. 服务器推送：主动推送资源
5. 请求优先级：优先加载重要资源
```

### 2.5 DNS 解析

```
DNS 解析过程：

1. 浏览器缓存
   ↓ 未命中
2. 操作系统缓存
   ↓ 未命中
3. 本地 DNS 服务器（ISP）
   ↓ 未命中
4. 根 DNS 服务器 → 返回顶级域名服务器地址
5. 顶级域名服务器 → 返回权威域名服务器地址
6. 权威域名服务器 → 返回 IP 地址

DNS 记录类型：
- A：域名 → IPv4
- AAAA：域名 → IPv6
- CNAME：别名 → 域名
- MX：邮件服务器
- NS：域名服务器
- TXT：文本记录

DNS 优化：
1. DNS 缓存
2. DNS 预解析（<link rel="dns-prefetch">）
3. DNS 负载均衡
```

### 2.6 网络安全

```
常见攻击：

1. XSS（跨站脚本攻击）
   - 注入恶意脚本
   - 防御：转义、CSP

2. CSRF（跨站请求伪造）
   - 伪造用户请求
   - 防御：Token、SameSite Cookie

3. SQL 注入
   - 注入 SQL 语句
   - 防御：参数化查询

4. 中间人攻击（MITM）
   - 窃听/篡改通信
   - 防御：HTTPS

5. DDoS（分布式拒绝服务）
   - 大量请求耗尽资源
   - 防御：限流、WAF、CDN

6. 重放攻击
   - 重放有效请求
   - 防御：时间戳、Nonce
```

**使用示例**：

```
   - 初始 cwnd = 1 MSS
   - 每个 RTT 翻倍增长
   - 直到达到 ssthresh 或出现丢包

   - 每个 RTT 增加 1 MSS
   - 线性增长，避免快速拥塞

3. 快重传（Fast Retransmit）
   - 收到 3 个重复 ACK
   - 立即重传丢失的报文
   - 不等待超时

4. 快恢复（Fast Recovery）
   - ssthresh = cwnd / 2
   - cwnd = ssthresh + 3
   - 进入拥塞避免阶段
```

### 验证题1：TCP 三次握手分析

**场景描述**：分析 TCP 连接建立过程中的报文交互，理解三次握手的工作原理。

**详细需求**：
1. 解释三次握手中每次握手的作用
2. 说明 SYN Flood 攻击的原理和防御方法
3. 分析为什么不是两次或四次握手

**测试用例**：
```
使用 Wireshark 抓取 TCP 三次握手过程：

1. 过滤条件：tcp.flags.syn == 1
2. 分析三次握手的报文：
   - 第一次：SYN=1, seq=x
   - 第二次：SYN=1, ACK=1, seq=y, ack=x+1
   - 第三次：ACK=1, seq=x+1, ack=y+1
```

**完整答案**：
```
1. 每次握手的作用：
   第一次：客户端请求建立连接，告知服务端自己的初始序号
   第二次：服务端确认收到请求，告知客户端自己的初始序号
   第三次：客户端确认收到服务端的确认，双方准备就绪

2. SYN Flood 攻击：
   原理：攻击者发送大量 SYN 但不完成第三次握手
   影响：服务端 SYN 队列满，无法处理正常请求
   防御：SYN Cookie、增加半连接队列、防火墙过滤

3. 为什么是三次：
   - 两次无法确认客户端的接收能力
   - 四次可以但效率低（ACK 和 SYN 可以合并）
   - 三次是建立双向可靠连接的最小次数
```

- [ ] 能说明 SYN Flood 攻击原理
- [ ] 能说明防御 SYN Flood 的方法

### 验证题2：HTTP 缓存机制分析

**场景描述**：分析 HTTP 缓存机制，理解强缓存和协商缓存的区别。

**详细需求**：
1. 说明强缓存和协商缓存的区别
2. 解释 Cache-Control 的常用指令
3. 说明 ETag 和 Last-Modified 的区别

**测试用例**：
```python
# 使用 requests 测试缓存
import requests

# 第一次请求
response1 = requests.get("https://example.com/api/data")
print(f"状态码: {response1.status_code}")
print(f"ETag: {response1.headers.get('ETag')}")
print(f"Last-Modified: {response1.headers.get('Last-Modified')}")

# 第二次请求（使用 If-None-Match）
headers = {"If-None-Match": response1.headers.get("ETag")}
response2 = requests.get("https://example.com/api/data", headers=headers)
print(f"状态码: {response2.status_code}")  # 应该是 304
```

**完整答案**：
```
1. 强缓存 vs 协商缓存：

   强缓存：
   - 不发送请求到服务器
   - 通过 Cache-Control 或 Expires 判断
   - 状态码 200（from cache）

   协商缓存：
   - 发送请求到服务器验证
   - 通过 Last-Modified 或 ETag 验证
   - 未修改返回 304，修改返回 200

2. Cache-Control 常用指令：
   - max-age=秒数：缓存有效期
   - no-cache：使用前需验证
   - no-store：不缓存
   - public：可被任何缓存存储
   - private：只能被浏览器缓存
   - must-revalidate：过期后必须验证

3. ETag vs Last-Modified：
   ETag：
   - 基于内容生成的唯一标识
   - 精确到内容变化
   - 计算成本较高

   Last-Modified：
   - 基于文件修改时间
   - 精确到秒
   - 无法检测内容未变但时间变化的情况
```

**自测检查清单**：
- [ ] 能说明强缓存和协商缓存的区别
- [ ] 能解释 Cache-Control 常用指令
- [ ] 能说明 ETag 和 Last-Modified 的区别

### 验证题3：HTTPS 安全分析

**场景描述**：分析 HTTPS 的安全机制，理解加密和证书验证过程。

**详细需求**：
1. 说明 HTTPS 如何防止中间人攻击
2. 解释证书链的验证过程
3. 分析 HTTPS 的性能开销

**测试用例**：
```python
# 使用 Python 检查 SSL 证书
import ssl
import socket

def check_ssl_cert(hostname):
    context = ssl.create_default_context()
    with socket.create_connection((hostname, 443)) as sock:
        with context.wrap_socket(sock, server_hostname=hostname) as ssock:
            cert = ssock.getpeercert()
            print(f"证书主题: {cert['subject']}")
            print(f"证书颁发者: {cert['issuer']}")
            print(f"有效期: {cert['notBefore']} - {cert['notAfter']}")

check_ssl_cert("www.baidu.com")
```

**完整答案**：
```
1. HTTPS 防止中间人攻击：
   - 证书验证：客户端验证服务器证书的真实性
   - 公钥加密：使用服务器公钥加密对称密钥
   - 消息认证：使用 MAC 验证数据完整性

2. 证书链验证过程：
   a. 服务器发送证书链
   b. 客户端验证叶子证书
   c. 逐级向上验证中间证书
   d. 最终验证到根证书（内置在浏览器/系统中）
   e. 验证每级证书的签名、有效期、域名

3. HTTPS 性能开销：
   - 握手开销：首次连接 1-2 个 RTT
   - 加密开销：CPU 计算量增加
   - 证书开销：证书传输占用带宽
   - 优化方案：Session Resumption、OCSP Stapling、HTTP/2
```
